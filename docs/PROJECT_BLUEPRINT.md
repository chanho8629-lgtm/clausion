## 1. 프로젝트 비전과 배경

### 1.1 왜 이 프로젝트를 만들었는가

직업훈련 교육 현장에서 강사는 평균 20~40명의 수강생을 동시에 관리합니다. 각 학생의 학습 상태를 파악하고 적절한 개입 시점을 판단하는 것은 인간 강사의 인지 한계를 넘어서는 일입니다.

**핵심 문제**:
- 학생이 "어디서 막혔는지" 스스로 인지하지 못하는 경우가 많음
- 강사가 개별 학생의 취약점을 파악하려면 상당한 시간과 에너지가 필요
- 상담은 있지만, 상담 전 준비와 상담 후 실행 관리가 체계적이지 않음
- 복습 시점과 방법이 학생 개인의 의지에만 의존

**해결 접근**: 학생 한 명 한 명에 대해 "디지털 트윈"을 만들어, AI가 학생의 학습 상태를 지속적으로 추론하고, 강사에게 실시간 인사이트를 제공하며, 학생에게는 개인화된 학습 처방을 제공합니다.

### 1.2 디지털 트윈이란

"디지털 트윈(Digital Twin)"은 원래 제조업에서 물리적 장비의 가상 복제본을 의미합니다. ClassPulse Twin은 이 개념을 교육에 적용합니다:

```
실제 학생 ←→ 디지털 트윈 (5차원 상태 모델)
              ├── 숙달도 (Mastery)
              ├── 실행력 (Execution)  
              ├── 동기 (Motivation)
              ├── 유지율 위험 (Retention Risk)
              └── 상담 필요도 (Consultation Need)
```

트윈은 학생의 활동 데이터(성찰일지, 코드 제출, 복습 완료율, 챗봇 대화, 게이미피케이션 등)를 종합하여 **하루 1회 자동 갱신**됩니다.

## 2. 아키텍처 설계 결정

### 2.1 기술 스택 선택 이유

| **Vite** | HMR 속도, ES Module 번들링. 개발 생산성 극대화 |
| **Railway** | Docker 기반 배포 자동화, PostgreSQL/Redis 관리형 서비스 |

### 2.2 모놀리식 vs 마이크로서비스

**결정: 모놀리식 (Modular Monolith)**

이유:
- 초기 팀 규모(1~3명)에서 마이크로서비스의 운영 오버헤드는 비효율적
- 도메인 모듈(ai/, domain/, api/, config/)로 논리적 분리는 유지
- AI 엔진들이 같은 데이터베이스를 공유해야 하므로 네트워크 호출 없이 직접 접근이 효율적
- 향후 분리가 필요하면 모듈 경계가 명확하여 쉽게 마이크로서비스로 전환 가능

### 2.3 비동기 처리 아키텍처

```
사용자 행동 (성찰 저장, 코드 제출 등)
    ↓
TwinInferenceDebouncer (Redis 5분 윈도우)
    ↓ (중복 제거)
AiJobService (@Async, 스레드 풀 4-16)
    ↓
AI 엔진 실행 (GPT-4o API 호출)
    ↓
AsyncJob 테이블에 결과 기록
    ↓
프론트엔드 폴링 (React Query refetchInterval)
```

**왜 5분 디바운싱인가?**
- 학생이 성찰일지 작성 → 2분 후 코드 제출 → 1분 후 복습 완료
- 디바운싱 없이는 3번의 GPT-4o API 호출 발생 (비용 + 불안정)
- 5분 윈도우로 묶으면 1번의 종합 추론으로 충분
- 실험 결과 **~80%의 불필요한 API 호출 절약**

## 3. AI 엔진 설계 철학

### 3.1 8개 AI 엔진 구조

ClassPulse Twin은 8개의 독립적 AI 엔진을 운영합니다. 각 엔진은 단일 책임 원칙(SRP)을 따릅니다:

| # | 엔진 | 역할 | 입력 | 출력 |
|---|------|------|------|------|
| 1 | **CurriculumAnalyzer** | 커리큘럼 분석 | 강의 자료 텍스트 + 학습 목표 | 원자 스킬, 선수 지식 그래프, 주별 핵심 개념, 오개념, 복습 포인트 |
| 2 | **QuestionGenerator** | 문항 생성 | 스킬 목록 + 난이도 | 평가 문항 (선택형/코딩형/서술형) |
| 3 | **TwinInferenceEngine** | 디지털 트윈 추론 | 8개 데이터 소스 통합 | 5차원 점수 + AI 인사이트 + 추세 |
| 4 | **ConsultationCopilot** | 상담 코파일럿 | 트윈 + 성찰 + 스킬 스냅샷 | 사전 브리핑 / 사후 요약 + 7일 액션 플랜 |
| 5 | **RecommendationAi** | 학습 추천 | 트윈 + 약점 스킬 | 개인화된 다음 단계 처방 |
| 6 | **ChatbotAi** | 학습 챗봇 | 대화 맥락 + 트윈 상태 | 트윈 인식 답변 |
| 7 | **CodeReviewAi** | 코드 리뷰 | 코드 + 약점 스킬 | 라인별 피드백 (트윈 연결) |
| 8 | **StudyGroupMatcherAi** | 스터디 매칭 | 학생 프로필 목록 | 상호보완적 그룹 편성 |

### 3.2 하이브리드 AI (규칙 + LLM) - 왜 이렇게 만들었는가

**순수 LLM만 쓰지 않는 이유:**
1. **비용**: 매번 GPT-4o를 호출하면 학생 100명 × 하루 1회 = 일 $15~30 발생
2. **일관성**: LLM은 같은 입력에 매번 다른 점수를 줄 수 있음
3. **설명 가능성**: "왜 이 점수인가?"에 규칙 기반 로직이 명확한 답을 제공
4. **장애 대응**: OpenAI API 다운 시에도 규칙 기반 점수는 계산 가능

**순수 규칙만 쓰지 않는 이유:**
1. 데이터 충돌 감지 불가 (예: 자신감은 높은데 코드 품질은 낮음 = 과신)
2. 자연어 인사이트 생성 불가 (강사에게 "이 학생은..." 설명이 필요)
3. 추세 해석 한계 (IMPROVING/STABLE/DECLINING 판단에 맥락 필요)

**하이브리드 접근:**
```
Step 1: 규칙 기반 점수 계산 (결정적, 무료, 즉시)
  mastery = confidence×0.30 + reviewRate×0.25 + codeQuality×0.25 + skillUnderstanding×0.20
  
Step 2: LLM 검증 및 보정 (확률적, 유료, 비동기)
  GPT-4o가 규칙 점수 + 원본 데이터를 보고:
  - 점수 조정 제안 (-10 ~ +10)
  - 데이터 충돌 감지
  - 한국어 인사이트 생성
  - 추세 방향 판단

Step 3: 안전 장치
  MAX_SCORE_DELTA = 25 (한 번에 25점 이상 변동 차단)
  → 이상치 방지, 학생/강사 혼란 예방
```

### 3.3 데이터 수집 전략 (8개 소스)

TwinDataCollector가 추론 전에 수집하는 데이터:

```
1. 성찰일지 (Reflections)
   → 자기 평가, 감정 상태, 막힌 지점, 자신감 점수
   → WHY: 학생의 주관적 인식은 객관적 지표만큼 중요

2. 복습 과제 (Review Tasks)  
   → 완료율, 예정 대비 실제 완료 비율
   → WHY: 실행력의 직접적 지표

3. 상담 (Consultations)
   → 빈도, 최근 상담 일자, 상태
   → WHY: 도움 요청 패턴이 자기주도성 판단 근거

4. 코드 분석 (Code Submissions)
   → 제출 수, 피드백 심각도 분포 (ERROR/WARNING/GOOD)
   → WHY: 실전 능력의 객관적 측정

5. 챗봇 대화 (Chatbot)
   → 메시지 수, 마지막 활동, 세션 수
   → WHY: 자기주도 학습 참여도 지표

6. 게이미피케이션 (Gamification)
   → 레벨, XP 획득 속도, 연속 출석일
   → WHY: 동기 부여 수준의 행동 지표

7. 스킬 스냅샷 (Skill Mastery Snapshots)
   → 스킬별 이해도/실습도/자신감/망각위험
   → WHY: 세분화된 역량 파악

8. 이전 트윈 상태 (Previous Twin)
   → 이전 점수, 추세
   → WHY: 이상치 탐지, 급격한 변화 감지
```

## 4. 핵심 기능별 설계 결정

### 4.1 간격 반복 복습 시스템 (ReviewScheduler)

**에빙하우스 망각 곡선 기반 설계:**

```
기본 간격: [0일, 1일, 3일, 7일, 14일, 30일]

트윈 인식 조정:
├── 망각위험 > 70% → 간격 ×0.5 (더 자주 복습)
├── 망각위험 50~70% → 간격 ×0.7
├── 망각위험 < 20% → 간격 ×1.5 (이미 숙달, 간격 늘림)
└── 이해도 < 40% → "긴급 복습" 태그 부여

일일 과제 수 제한:
├── 위험도 높음 (overall_risk > 70) → 최대 5개
├── 위험도 중간 → 최대 3개
└── 위험도 낮음 → 최대 2개
```

**왜 이렇게 설계했는가:**
- 연구 근거: 에빙하우스 망각 곡선은 100년 이상 검증된 이론
- 개인화: 일률적 간격이 아닌, 학생의 실제 상태에 맞춘 적응형 간격
- 인지 부하 관리: 어려워하는 학생에게 과제를 더 많이 주되, 한도를 둠
- 자기 결정: 학생이 직접 "완료" 체크하여 자기주도 학습 습관 형성

### 4.2 상담 코파일럿 (ConsultationCopilot)

**상담 전 브리핑 (Pre-Briefing):**
```json
{
  "studentSummary": "김OO 학생은 Python 기초 과정에서...",
  "weakSkills": ["반복문 중첩", "리스트 컴프리헨션"],
  "recentReflections": "최근 성찰에서 '어디서부터 틀렸는지 모르겠다'고 기록",
  "suggestedQuestions": [
    "반복문을 작성할 때 어떤 순서로 생각하나요?",
    "리스트 컴프리헨션과 for문의 차이를 설명해볼 수 있나요?"
  ],
  "riskAlerts": ["자신감 급락 (73→41)", "3일 연속 복습 미완료"]
}
```

**왜 브리핑이 필요한가:**
- 강사는 하루에 여러 학생을 상담함 → 각 학생 맥락을 기억하기 어려움
- AI가 트윈 데이터를 요약해주면 **상담 준비 시간 70% 절감**
- 데이터 기반 질문으로 상담 품질 향상

**상담 후 요약 (Post-Summary):**
```json
{
  "summary": "학생은 반복문 개념은 이해하나, 중첩 시 인덱스 관리에서 혼란...",
  "causeAnalysis": "시각적 사고에 강한 학생으로, 추상적 인덱스 추적이 어려움",
  "actionPlan": [
    {"day": 1, "task": "반복문 시각화 도구로 step-by-step 추적 연습"},
    {"day": 3, "task": "간단한 2중 반복문 3문제 풀기"},
    {"day": 7, "task": "리스트 컴프리헨션으로 동일 문제 변환"}
  ],
  "followUpDate": "2026-04-20",
  "courseRecommendation": "시각화 중심 학습 자료 추가 권장"
}
```

**왜 7일 액션 플랜인가:**
- 너무 길면 실행 의지 저하, 너무 짧으면 행동 변화 불가
- 7일은 상담-실행-점검 사이클에 최적
- 각 일자별 구체적 과제로 모호함 제거

### 4.3 과정별 데이터 분리

**결정: 모든 데이터는 courseId로 필터링**

```
학생 A가 "Python 기초"와 "데이터 분석" 2개 과정 수강 시:
├── 트윈 상태 → 과정별 독립 (Python 트윈 ≠ 데이터분석 트윈)
├── 복습 과제 → 과정별 독립
├── 성찰일지 → 과정별 독립
├── 게이미피케이션 → 과정별 독립
└── 상담 기록 → 과정별 독립
```

**왜 이렇게 만들었는가:**
- Python에서 잘하는 학생이 데이터 분석에서는 어려울 수 있음
- 과정이 합쳐지면 "어떤 과정에서 약한지" 구분 불가
- 강사는 자신이 담당하는 과정의 학생 상태만 볼 수 있어야 함
- 사이드바에서 과정을 선택하면 모든 대시보드 데이터가 해당 과정으로 필터링됨

### 4.4 역할 기반 접근 제어 (RBAC)

**3계층 역할 설계:**

```
STUDENT (학생)
├── 자신의 트윈 상태 조회
├── 성찰일지 작성/조회
├── 복습 과제 확인/완료
├── 코드 제출 및 AI 피드백
├── 학습 챗봇 이용
├── 상담 요청 및 참여
├── 스터디 그룹 참여
└── 개인화된 학습 추천 확인

INSTRUCTOR (강사)
├── 과정 생성/관리
├── 커리큘럼 업로드 → AI 분석 트리거
├── 학생 목록 + 트윈 레이더 차트
├── AI 생성 문항 검토/승인
├── 상담 스케줄링 + AI 브리핑
├── 위험 학생 알림
└── 수강생 성과 분석

OPERATOR (운영자)
├── 전체 과정 관리
├── 위험 학생 전체 조회 (과정 횡단)
├── 개입 센터 (조기 경보 + 조치)
├── 강사 성과 분석
├── 출석 관리
├── 감사 로그
├── What-if 시뮬레이션
└── 운영 리포트 생성
```

**왜 ADMIN이 아닌 OPERATOR인가:**
- "관리자"가 아닌 "운영자" — 기술 시스템 관리가 아닌 **교육 운영** 관점
- 운영자는 교육 프로그램의 품질과 학생 성과에 집중
- 시스템 관리(서버, DB)는 개발팀의 영역으로 분리

### 4.5 실시간 기능 설계

| 기능 | 기술 | 이유 |
|------|------|------|
| 알림 | SSE (Server-Sent Events) | 단방향 푸시면 충분. WebSocket보다 가볍고 자동 재연결 |
| 그룹 채팅 | WebSocket + STOMP + RabbitMQ | 양방향 실시간 필요. MQ로 메시지 유실 방지 |
| 화상 상담 | LiveKit (WebRTC) | 오픈소스, 셀프 호스팅 가능, 화면 공유 + 녹화 |

### 4.6 게이미피케이션 시스템

```
XP 획득 이벤트:
├── 성찰일지 작성 → +10 XP
├── 복습 완료 → +15 XP  
├── 코드 제출 → +20 XP
├── 상담 참여 → +25 XP
└── 연속 출석 보너스 → 3일: +10, 7일: +30, 30일: +100

레벨 체계:
├── Lv.1 새싹 (0 XP)
├── Lv.2 탐험가 (100 XP)
├── Lv.3 도전자 (300 XP)
├── Lv.4 실력자 (600 XP)
└── Lv.5 마스터 (1000 XP)

배지: 특별 달성 조건 (첫 성찰, 7일 연속, 전 과목 복습 완료 등)
```

**왜 게이미피케이션을 넣었는가:**
- 직업훈련생의 평균 연령대(20~30대)에 게이미피케이션 효과가 검증됨
- 외적 동기(XP, 레벨)가 내적 동기(학습 자체의 즐거움)로 전환되는 가교 역할
- 트윈 동기 점수(Motivation)에 게이미피케이션 참여도가 반영됨

## 5. 데이터 모델 설계 원칙

### 5.1 트윈 데이터의 시간축 관리

```
student_twin (현재 상태 - 최신 스냅샷)
  → 항상 1개만 존재 (student_id + course_id UNIQUE)
  → 조회 성능 최적화

twin_score_history (이력 - 시계열)
  → 매 추론마다 새 레코드
  → 추세 분석, 차트 표시용

skill_mastery_snapshot (스킬별 상세)
  → 추론 시점마다 각 스킬의 스냅샷
  → 강사가 "이 학생의 어떤 스킬이 약한가" 확인
```

**왜 이렇게 분리했는가:**
- 현재 상태 조회(대시보드)는 가장 빈번한 쿼리 → 단일 row로 O(1) 조회
- 이력 조회(그래프)는 가끔 필요 → 별도 테이블에서 시계열 조회
- 과거 데이터를 삭제하지 않고 축적 → 장기 추세 분석 가능

### 5.2 JSONB 활용 전략

일부 데이터는 정규화 대신 JSONB 컬럼을 사용합니다:

| 컬럼 | 테이블 | 이유 |
|------|--------|------|
| `briefingJson` | consultations | AI 응답 구조가 유동적, 스키마 변경 없이 확장 가능 |
| `actionPlanJson` | consultations | 7일 계획은 상담마다 다른 구조 가능 |
| `emotionSummary` | reflections | 감정 분석 결과가 string 또는 object일 수 있음 |
| `aiAnalysisJson` | reflections | AI 분석 포맷이 진화 중 |
| `dataConflictsJson` | student_twin | 데이터 충돌 목록의 개수가 가변적 |

**원칙**: 쿼리 조건으로 자주 사용되는 데이터 → 정규 컬럼. AI 생성 결과나 유동적 구조 → JSONB.

### 5.3 마이그레이션 전략 (Flyway)

```
V1__init_schema.sql      → 핵심 테이블 (users, courses, enrollments)
V2__v2_features.sql      → 확장 기능 (reflections, reviews, consultations)
V3__seed_badges.sql      → 초기 데이터 (배지 목록)
V4__action_plans.sql     → 액션 플랜 테이블
V6__twin_enhanced.sql    → 트윈 v2 (스킬 스냅샷, 이력)
...
```

**왜 Flyway인가:**
- 스키마 변경을 코드로 추적 → Git에서 이력 관리
- 프로덕션 배포 시 자동 마이그레이션
- `ddl-auto: validate`로 코드-DB 불일치 즉시 감지

## 6. 프론트엔드 설계 원칙

### 6.1 컴포넌트 구조

```
pages/           → 라우트 단위 페이지 (데이터 페칭 + 레이아웃)
components/      → 재사용 가능한 UI 블록
  ├── student/   → 학생 전용 (TwinStateCard, ReviewTimeline...)
  ├── instructor/→ 강사 전용 (StudentTwinCard, RiskHeatmap...)
  ├── common/    → 공용 (Modal, ProgressBar, RadarChart...)
  └── layout/    → 레이아웃 (Sidebar, AppShell...)
api/             → 서버 통신 (Axios 기반, 1 파일 = 1 도메인)
store/           → 전역 상태 (Zustand, 4개 스토어)
hooks/           → 커스텀 훅 (데이터 페칭, 상태 관리)
```

### 6.2 상태 관리 전략

```
서버 상태 (React Query)
├── 트윈 데이터, 복습 과제, 성찰 목록, 상담 기록
├── queryKey에 courseId 포함 → 과정 변경 시 자동 캐시 무효화
└── staleTime 설정으로 불필요한 재요청 방지

클라이언트 상태 (Zustand)
├── authStore: 로그인 사용자, 토큰 (localStorage 영속)
├── courseStore: 수강 과정 목록, 현재 선택 과정
├── sidebarStore: 사이드바 접힘 상태
└── chatbotStore: 챗봇 모달 표시 여부
```

**왜 Redux가 아닌 Zustand인가:**
- 서버 상태는 React Query가 담당 → 클라이언트 상태만 관리하면 됨
- 4개의 작은 스토어만 필요 → Redux의 보일러플레이트는 과도
- localStorage 연동이 내장되어 있어 인증 상태 영속화가 간단

### 6.3 UI/UX 설계 원칙

- **glassmorphism + gradient**: 모던하면서 교육 플랫폼에 적합한 느낌
- **Framer Motion**: 데이터 로딩/전환 시 자연스러운 애니메이션
- **Tailwind CSS**: 일관된 디자인 시스템, 다크모드 대응 용이
- **반응형**: max-w-5xl 기준, 모바일-태블릿-데스크톱 대응
- **접근성**: useReducedMotion 훅으로 모션 감도 조절

## 7. 보안 설계

### 7.1 인증/인가

```
JWT 기반 인증:
├── 로그인 → JWT 토큰 발급 (24시간 만료)
├── 매 요청 → Authorization: Bearer <token>
├── JwtAuthenticationFilter → 토큰 검증 + SecurityContext 설정
└── SecurityConfig → 역할별 엔드포인트 접근 제어

역할 기반 경로 제어:
├── /api/student/** → STUDENT 역할만
├── /api/instructor/** → INSTRUCTOR 역할만
├── /api/operator/** → OPERATOR 역할만
└── /api/auth/**, /api/health → 전체 허용
```

### 7.2 데이터 보호

- 비밀번호: BCrypt 해싱
- CORS: 허용 origin 명시적 지정
- SQL Injection: JPA Parameterized Query 사용
- XSS: React의 기본 이스케이핑 + Content-Type 헤더
- HTTPS: Railway 자동 SSL
